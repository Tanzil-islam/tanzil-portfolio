Fine-Tuning BanglaBERT on a Multimodal Bengali Dataset — What Broke and How I Fixed It

A hands-on account of building a Bengali sentiment & intent classifier using Transformer models — including the frustrating performance issues nobody warns you about.

Tanzil Islam March 2026 BanglaBERT · mBERT · XLM-RoBERTa

Bengali is the 7th most spoken language in the world — with over 230 million speakers — yet it remains one of the most underserved languages in NLP research. When I took on a project to build a multimodal sentiment and intent classifier for Bengali customer voice and text data, I quickly discovered just how thin the tooling support really is.

This article is a full technical walkthrough of the system I built: the model choices, the architecture, the training pipeline — and more importantly, the three serious performance problems I hit, what caused them, and exactly how I fixed each one. If you're working on Bengali NLP or low-resource language modeling in general, I hope this saves you the weeks it cost me.

Why Bengali NLP is Hard

Bengali script (বাংলা) is an abugida — each consonant carries an inherent vowel that can be modified by diacritics. This creates morphological complexity that Latin-script tokenizers were never designed to handle. A single Bengali word can encode what English might express in a full phrase.

Standard WordPiece tokenizers, trained on English-dominant corpora, decompose Bengali words into meaningless character fragments. In my initial experiments, over 62% of tokens in Bengali-only test samples resolved to [UNK] using a standard BERT tokenizer. You simply cannot learn semantics from a sequence of unknown tokens.

The core challenge

Bengali has ~120,000 unique word forms when accounting for inflection and agglutination. Standard BERT vocabulary covers fewer than 800 Bengali subword units — a coverage gap that makes naive fine-tuning nearly useless.

Choosing the Right Model

I evaluated four model backbones before settling on my final architecture. The decision had to factor in inference speed, fine-tuning stability on a relatively small dataset (~18,000 labeled samples), and the ability to handle mixed-script input (Bengali + occasional English code-switching).

The Architecture: Multimodal Text + Voice

The dataset contained two modalities: Bengali text messages and voice recordings of customer calls. The goal was to classify each interaction into one of five intent categories: complaint, inquiry, feedback, escalation, and resolution.

I chose a late-fusion architecture — each modality encoded independently, then a cross-attention layer merges them before the classification head. More flexible than early fusion, and allows text-only or audio-only inference.

The projection mismatch problem

BanglaBERT outputs 768-dimensional embeddings. Wav2Vec2-base outputs 512-dimensional. When I concatenated them after projecting both to a shared 256-dim space, the text embeddings dominated every gradient update. By epoch 5, audio was contributing less than 4% to the final prediction.

Problem 1: Overfitting on Bengali Fine-Tuning Data

After epoch 4, training accuracy climbed toward 94% while validation accuracy plateaued at 68% and began declining. A 26-point train-val gap on 18,000 samples.

Standard fixes — dropout (0.3), weight decay (0.01), early stopping — made no meaningful difference. Then I visualized attention weights using BertViz. The model was routing almost all attention through the [CLS] token, ignoring contextual Bengali tokens entirely. It had found a memorization shortcut.

The fix: selective layer unfreezing + stochastic depth

I froze the bottom 8 BanglaBERT layers and only fine-tuned layers 9–12 plus the pooler. This preserved the Bengali language representations in the lower layers while allowing the upper layers to adapt to intent classification.

Python — selective layer freezing
for name, param in model.banglabert.named_parameters():
    # Freeze bottom 8 layers, keep top 4 trainable
    layer_num = extract_layer_num(name)
    if layer_num is not None and layer_num < 9:
        param.requires_grad = False

# Stochastic depth — drop encoder layers probabilistically
drop_path_rates = torch.linspace(0.0, 0.1, 12)
for i, layer in enumerate(model.banglabert.encoder.layer):
    layer.drop_path_rate = drop_path_rates[i].item()

I also added a token-level auxiliary loss to force the model to attend to the full Bengali sequence. Train-val gap collapsed from 26% to 7%. Validation F1 improved from 0.68 to 0.86.

Problem 2: Bengali Subword Boundary Errors

Even with BanglaBERT's superior vocabulary, hasanta (্) and matra (া, ি, ী...) characters were being separated from their host consonants during subword segmentation. This broke the phonological integrity of tokens.

The word করতেছি (am doing) was split into কর + তে + ছি — valid-looking splits that destroyed the morphological unit the model needed to understand aspect and tense.

Python — Bengali Unicode normalization
import unicodedata
import re

def normalize_bengali(text: str) -> str:
    # NFC: compose base character + combining diacritic → single code point
    text = unicodedata.normalize('NFC', text)
    
    # Remove zero-width non-joiner if not between consonants
    text = re.sub(r'(?<![ক-হ়-৿])\u200C(?![ক-হ়-৿])', '', text)
    
    # Normalize Bengali numerals to ASCII
    bn_digits = str.maketrans('০১২৩৪৫৬৭৮৯', '0123456789')
    return text.translate(bn_digits)

Problem 3: Inference Latency in Production

After resolving training issues, I had BanglaBERT + Wav2Vec2 fused via cross-attention, Bengali F1 of 0.86. But inference was taking 1.8 seconds per request. Our SLA was 400ms.

The optimization path

PyTorch Profiler showed 64% of latency in BanglaBERT's self-attention layers 1–4 — high redundancy in attention patterns on Bengali sequences. My three-step fix: INT8 dynamic quantization on linear layers; pruning 4 of 12 encoder heads using Taylor importance scores; Flash Attention 2 reducing memory I/O by ~40%.

Key Takeaways

1. Tokenizer first, model second

BanglaBERT's SentencePiece vocabulary gave 97% coverage — nothing else was close. The tokenizer choice matters more than the architecture for Bengali.

2. Always apply NFC normalization before tokenization

Bengali Unicode has multiple valid encodings of the same character. Without NFC, the same word produces different token sequences depending on how it was typed.

3. Visualize attention before diagnosing overfitting

My problem was [CLS] token collapse — an architectural issue, not a regularization one. Dropout and weight decay would never have fixed it.

4. Multimodal fusion requires active modality balancing

Without gradient scaling and a learnable modality gate, text will drown audio within the first few epochs — silently.

5. Profile before optimizing

PyTorch Profiler found the real bottleneck in 20 minutes. I spent 3 days optimizing the wrong layers before that.
